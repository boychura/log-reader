## Goal

Build a single-page React + TypeScript web app that lets users paste or upload log files, displays them in a clean monospaced log viewer with line numbers, search, log-type filters, copy/clear controls, dark mode by default, and per-tag color highlighting (`[error]`, `[warning]`, `[info]`, `[debug]`, `[success]`, `[critical]`).

## Context

- Workspace is empty (`log-reader/`) except for `.gitignore` and the planning session folder. No existing source files.
- The user requested a simple frontend stack: React, TypeScript, HTML, CSS. No backend, no SSR, no persistence required.
- Default theme must be dark mode. Layout must remain comfortable for long reading sessions and responsive across viewport widths.
- Tag detection rule: `[<token>]` at the start of a line (after optional whitespace). Case-insensitive match. Lines without a known tag are rendered as plain logs.
- Tag-to-color mapping is fixed by the spec:
  - `[error]` — red
  - `[warning]` — orange/yellow
  - `[info]` — blue
  - `[debug]` — gray
  - `[success]` — green
  - `[critical]` — dark red / purple
- Lines preserve original spacing; only the leading tag gets a chip-style color treatment and the row gets a subtle background tint.
- The deliverable is the complete working web app code, plus install/run instructions.

## Assumptions

- Node.js 20.x and npm 10.x are available in the environment (Vite 5 requires Node ≥18).
- Build tooling: Vite 5 with the `@vitejs/plugin-react` plugin. Test runner: Vitest (ships with Vite).
- All log processing is done client-side; no API calls.
- "File upload" accepts `.log` and `.txt` via a `<input type="file">` plus drag-and-drop on the input panel. Max file size enforced at 25 MB to keep memory bounded.
- "Search" is a case-insensitive substring match against the visible (post-filter) line text; "Filter by log type" is a multi-select toggle (default = all enabled) on the six known tags.
- "Copy visible logs" copies the currently rendered (post-search, post-filter) lines — preserving their original text — to the clipboard via `navigator.clipboard.writeText`.
- Dark mode is the only shipped theme; a light-mode toggle is out of scope.
- No persistence between page reloads (acceptable per "preserve original log formatting and spacing" — no requirement to remember state).

## Implementation Notes

### Approach (chosen)

Vite 5 + React 18 + TypeScript SPA. Rationale:
- Matches the user's "simple frontend stack such as React, TypeScript, HTML, CSS" requirement.
- Vite gives instant dev start, zero config for TS/JSX, and ships Vitest so we can add unit tests without extra setup.
- Pure client-side — no API, no SSR. The 12 features map cleanly to one component tree, one parser module, and one filter selector module.

Alternatives considered and rejected:
- Next.js: adds routing/SSR overhead with no benefit; this is a single-screen viewer.
- Plain HTML + JS: doesn't match the user's React + TS request and weakens type safety on tag detection.

### Type contracts (locked-in for cross-file consistency)

```ts
// src/types/log.ts
export type LogType =
  | 'error' | 'warning' | 'info' | 'debug' | 'success' | 'critical' | 'unknown';

export interface LogEntry {
  index: number;        // 1-based original line number (never renumbered by filter)
  rawText: string;      // exact original line content (no trimming of inner spaces)
  tag: LogType;         // detected tag at start of line, else 'unknown'
  content: string;      // rawText with the leading "[tag] " stripped when tag !== 'unknown'
}
```

```ts
// src/constants/logTypes.ts
export const KNOWN_TAGS: ReadonlySet<LogType> = new Set([
  'error', 'warning', 'info', 'debug', 'success', 'critical',
]);

export const LOG_TYPE_ORDER: readonly LogType[] = [
  'error', 'warning', 'info', 'debug', 'success', 'critical',
];

export const TAG_LABEL: Record<LogType, string> = {
  error: '[error]',
  warning: '[warning]',
  info: '[info]',
  debug: '[debug]',
  success: '[success]',
  critical: '[critical]',
  unknown: '',
};
```

### Parser rules (parser.ts)

- Split input on `\n` only (do NOT split on `\r`); keep trailing empty lines produced by a final newline.
- For each line: strip leading whitespace, then test `^\[([a-zA-Z]+)\]` against the stripped prefix.
- Compare captured token (lowercased) against `KNOWN_TAGS` — match is case-insensitive.
- If matched: `tag = token`, `content = line.replace(/^\s*\[[a-zA-Z]+\]\s?/, '')`.
- Else: `tag = 'unknown'`, `content = line` (preserves all whitespace/tabs).
- Assign `index` as the 1-based position in the source array (so filtering never changes the displayed number).

### Filter rules (filter.ts)

```ts
export function applyFilters(
  entries: LogEntry[],
  query: string,
  active: ReadonlySet<LogType>,
): LogEntry[] {
  const q = query.trim().toLowerCase();
  return entries.filter((e) => {
    if (!active.has(e.tag)) return false;
    if (q && !e.rawText.toLowerCase().includes(q)) return false;
    return true;
  });
}
```


### Rendering rules (LogLine.tsx)

- Wrapper: `<div className={`log-line log-line--${entry.tag}`}>`
  - When `tag === 'unknown'`, omit the modifier class (CSS targets `.log-line` for the base style only).
  - When tag is known, the modifier class sets `background-color: var(--bg-${tag})` via theme.css.
- Inside the wrapper:
  - `<span className="log-line__gutter">{entry.index}</span>` — sticky left, right-aligned, tabular numerals.
  - `<span className={`log-tag log-tag--${entry.tag}`}>{TAG_LABEL[entry.tag]}</span>` — pill with `color: var(--fg-${tag})`, monospaced.
  - `<span className="log-line__content">{entry.content}</span>` — preserves whitespace via `white-space: pre-wrap`.

### Risks & mitigations

- **Large file memory pressure** — cap file size at 25 MB in `LogInputPanel.handleFile`. Show an inline error if exceeded.
- **Clipboard API unavailable** (e.g., non-secure context) — fall back to a hidden `<textarea>` + `document.execCommand('copy')` if `navigator.clipboard` is missing; show "Copy failed" toast on rejection.
- **Mixed `\r\n` line endings** — normalize by stripping trailing `\r` from each parsed line, but never strip whitespace inside the line body.
- **Tag case variants** — `[ERROR]` and `[Error]` must both render the red chip. The parser already lowercases the captured token before lookup.


## Out of Scope

- Light mode toggle (dark mode is the only shipped theme).
- Persisting logs, search, or filter state across reloads (no `localStorage` / IndexedDB).
- Server-side log ingestion, streaming, or websocket tailing.
- Authentication, multi-user, sharing, or cloud sync.
- Custom regex-based rules for unknown log formats (only the six named tags are colored).
- Exporting filtered logs to file (`.log` / `.txt` download). Only "Copy visible logs" to clipboard is in scope.
- Virtualization for >100k lines (acceptable performance target is up to ~50k lines; above that, plain rendering may slow but must not crash).
- Internationalization of UI strings (English only).
- Mobile-native packaging (Capacitor / Tauri); the web app is responsive but not packaged.
- Browser extensions, service workers, or PWA offline support.
